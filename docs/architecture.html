Data Flow — Filtered Request

01

User interacts with a filter

The FilterBar component updates shared state through the useFilters composable (e.g. warehouse, category, status, month).

02

View triggers an API call

A view (e.g. Dashboard.vue) watches filter changes and calls api.js, which uses Axios to send a GET request with the active filters as query params.

03

FastAPI receives the request

The route handler in main.py reads optional query params, treating 'all' as "no filter," and filters the in-memory data sequentially.

04

Pydantic validates the response

The route's response_model (e.g. List[InventoryItem]) validates and serializes the result. Invalid records surface as 500s during development.

05

Client stores raw data in refs

The view stores results in reactive refs (e.g. allOrders, inventoryItems). Derived values — totals, groupings, chart series — live in computed().

06

Template re-renders

Vue's reactivity updates tables, summary cards, and custom SVG charts. Modal components open with detail records via props + events.

Key Design Decisions

No Database
All data loaded once from JSON at startup. Filtering happens in Python — fine for the demo's data volume.
Server-side Filters
Four shared filters (warehouse, category, status, month) passed as query params. 'all' means "skip this filter."
Reactive Pattern
Raw API results in ref(), derived values in computed(). Composables share filter state across views.
Validation
Pydantic models validate every response. Any drift between JSON files and models surfaces immediately.
Production Gaps
No auth, CORS open to all origins, no rate limiting. Demo-only — not production-ready.
